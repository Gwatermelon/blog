# 技术博客编写规范

本文件是仓库内大模型与 Agent 的内容编写入口。创建或实质性改写技术博客时，必须先阅读本文件，再开始拟稿。

> 维护说明：具体要求全部写成独立的“规则卡片”。新增要求时复制一张卡片；删除要求时删除整张卡片；临时停用时把 `状态` 改为 `停用`。

## 1. 适用范围

- 适用于 `content/` 下技术博客的创建、扩写、重写和技术性校对。
- 用户在当前任务中的明确要求优先于本文件；发生冲突时，先指出冲突，再按用户要求执行。

## 2. 写作前必读

开始撰写前，依次读取：

1. 本文件；
2. 目标栏目对应的 `docs/templates/*.md`；
3. 同栏目中 1～2 篇现有文章，用于确认仓库实际采用的格式和术语。

模型或 Agent 不需要阅读 `docs/CONTENT_AUTHORING_GUIDE.md`，该文件仅供人工作者参考。

如果这些文件互相冲突，按“当前用户要求 → 本文件 → 栏目模板 → 既有文章”的顺序处理。

## 3. 规则执行方式

- 只执行 `状态: 启用` 且适用范围命中当前文章的规则。
- `必须` 表示不满足就不能交付；`建议` 表示应优先遵守，但可根据主题合理调整。
- 每条规则只表达一个核心要求，避免把互不相关的要求写在同一张卡片中。
- 规则 ID 一经使用尽量不要改名；建议按类别递增编号，便于讨论和追踪。
- 交付前根据每张启用规则中的“验收”逐项自检。

推荐的规则 ID 前缀：

| 前缀 | 类别 |
| --- | --- |
| `META` | 选题、读者与写作目标 |
| `STRUCT` | 文章结构 |
| `STYLE` | 语言与表达 |
| `TECH` | 技术准确性与术语 |
| `CODE` | 示例代码与命令 |
| `SOURCE` | 引用、链接与事实来源 |
| `MEDIA` | 图片、表格与公式 |
| `SEO` | 标题、摘要与 Front Matter |
| `QA` | 校对、构建与交付检查 |

## 4. 规则卡片模板

新增规则时复制以下完整区块，并替换尖括号中的内容。

```markdown
### [<类别>-<三位编号>] <规则名称>

- 状态: 启用
- 级别: 必须
- 适用: 全部技术博客
- 指令: <用一句或几句可执行的话描述模型必须做什么。>
- 验收: <描述如何客观判断该规则已经满足。>
- 例外: <允许不执行的明确场景；没有则写“无”。>
```

字段约定：

- `状态`：只能写 `启用` 或 `停用`。
- `级别`：只能写 `必须` 或 `建议`。
- `适用`：可写 `全部技术博客`，也可限定目录、栏目、语言或文章类型。
- `指令`：使用明确动作，例如“解释”“验证”“引用”“避免”，不要只写抽象目标。
- `验收`：应尽量可检查，避免“质量高”“足够好”等主观表述。
- `例外`：明确规则边界，防止模型机械套用。

## 5. 当前规则

下面是建议的起始规则。可直接修改、停用或删除。

### [STRUCT-001] 使用主题驱动的结构

- 状态: 启用
- 级别: 建议
- 适用: 全部技术博客
- 指令: 如果提供的撰写文档有明确主题结构优先使用文档结构，否则按照“问题或背景 → 核心原理 → 示例或推导 → 局限与误区 → 总结”组织内容，并根据主题删减不适用的章节。
- 验收: 各章节围绕核心问题递进，且不存在只有标题、提示语或占位文本的章节。
- 例外: 教程、故障复盘、论文解读等已有更合适叙事顺序的文章。


### [MEDIA-001] 图片处理

- 状态: 启用
- 级别: 必须
- 适用: 全部技术博客
- 指令: 提供的文章若涉及到粘贴的公式相关的图片，提取图片内容写成机打公式。
- 验收: 图片内容完整提取成机打公式。
- 例外: 图片不是文本表述，而是模型结构图，统计图时，不需要提取内容。

### [MEDIA-002] 避免公式被 Markdown 误解析

- 状态: 启用
- 级别: 必须
- 适用: 全部技术博客
- 指令: 撰写 $$...$$ 展示公式时，不得让等号或减号单独占据一行，因为它们可能被 Markdown 识别为 Setext 标题分隔符并拆散公式。连续等式应把运算符与前后表达式写在同一行，或使用 aligned、split 等 LaTeX 环境组织换行。
- 验收: 源文件的展示公式中不存在内容仅为等号或减号的行；Hugo 构建后的文章 HTML 中不存在由公式分隔符产生的 h1～h6 标签，例如公式起始分隔符被包进 h1 标签。
- 例外: 等号或减号位于代码块、普通正文或 Markdown 表格分隔行中，且不处于展示公式内部。
